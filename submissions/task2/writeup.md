# Task 2 Writeup

A local-file audit log is weak for production because it is tied to one filesystem instead of the shared deployment control plane. A concrete failure mode is an operator running the CLI in an ephemeral CI container: the MLflow alias update succeeds, the container is later deleted, and the rollback history disappears with `promotion-log.jsonl`. It also has no real concurrency control, so two operators can promote at nearly the same time and leave an audit history that does not accurately describe the final alias state. For production use, I would add a `--dry-run` / preflight mode that prints the current alias target, the exact target MLflow version, key metrics, and the audit event that would be written before making any mutation. That gives operators a cheap way to catch accidental config IDs or stale registry state before changing `production`.
